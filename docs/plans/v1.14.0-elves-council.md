# v1.14.0 Elves Council

Branch: `codex/v1.14.0-elves-council`

## Mission

Add a chat-native Elves Council workflow to the Elves skill. The council should let a user type
`/council`, `/ec`, or `/elves-council` in Codex or Claude Code and get a fast, read-only,
native-subagent synthesis of a design, planning, review, or debugging question.

The release is done when the Claude skill surface (`SKILL.md`), the Codex surface (`AGENTS.md`),
the README, config examples, reference docs, changelog, and consistency checks all describe the
same v1.14.0 council model: native subagents first, external providers optional, quick/read-only by
default, and one decisive synthesized answer rather than five visible reports.

## Context

This feature is inspired by Fable-style harness engineering: task routing, role-specific experts,
bounded reports, and synthesis. It is not a model identity, persona clone, safety-policy clone, or
replacement for the Elves run loop.

The existing math module already proves the underlying pattern:

- role slots instead of hardcoded model names;
- independent scouts and critics;
- synthesis after fan-out;
- ledgers for material model/reviewer calls;
- clear separation between generated advice and verified results.

Council generalizes the useful parts for ordinary software and planning work, while keeping the
happy path much lighter than the math workflow.

## Product Principles

- **Quick Council is the product.** Default `/council <task>` must feel like asking a few sharp
  colleagues for a synthesized answer, not like starting an Elves run.
- **Native first.** Use Codex subagents in Codex and Claude Code subagents in Claude Code. External
  providers belong to optional Deep Council mode, not the default path.
- **Read-only by default.** Quick Council inspects and recommends. It does not edit files, create
  branches, open PRs, install packages, or mutate run state.
- **One answer, visible dissent.** The synthesis leads with a recommendation, then names the
  strongest disagreement, risk, and next action. Raw role reports are hidden unless requested.
- **Adaptive, not ceremonial.** Default to two or three roles. Do not convene five agents for a
  small question.
- **No debate loop in v1.** Role agents work independently; the synthesizer reconciles them.
- **No Fable bulk.** Do not import vendor identity, policy framing, persona mechanics, or
  consumer-app guardrails.

## Scope

### In Scope

- New council workflow and prompt reference docs.
- Skill surface updates for both Claude-compatible agents (`SKILL.md`) and Codex (`AGENTS.md`).
- README, config, tool-config examples, changelog, and release metadata updates.
- Consistency checks for council aliases, native-first defaults, read-only Quick Council, optional
  Deep Council, and no Fable identity/policy leakage.
- Native-subagent operating guidance for Codex and Claude Code.
- Run Council logging guidance that reuses the existing execution log and `.elves-session.json`
  patterns when a council call happens inside an existing Elves run.

### Out of Scope

- A real slash-command parser or hosted command runtime.
- Mandatory OpenRouter or external provider calls.
- Multi-turn debate between subagents.
- Dynamic AI-invented roles.
- Automatic code edits from `/council`.
- Separate PR/session scaffolding, council-specific ledger system, or provider framework for normal
  council calls.
- Rich terminal or web UI for council reports.
- Copying Fable system prompt identity, policy, or safety sections.

## Proposed User Experience

```text
/council should we migrate this auth middleware to the new request context API?
/ec review the plan for brittle assumptions
/elves-council --roles architect,tester think through the export rewrite
/council --verbose what could break if we change this utility?
```

Default output shape:

```markdown
Recommendation: [take a position]

Why:
- [evidence]
- [evidence]

Dissent / risks:
- Skeptic: [strongest objection]
- Tester: [verification gap]

Next actions:
1. [specific action]
2. [specific action]
```

Supported v1 command semantics to document:

- `/council <task>`: Quick Council, native/read-only/default.
- `/ec <task>`: short alias for Quick Council.
- `/elves-council <task>`: explicit alias.
- `--brief`: two-role fast pass.
- `--verbose` or `--show-reports`: include individual reports.
- `--json`: return structured report for harness use.
- `--roles a,b,c`: override selected roles.
- `--n N`: cap role count.
- `--run`: attach the council result to an existing Elves run log.
- `--deep`: reserved/optional external-provider mode; should degrade gracefully when unconfigured.

## Architecture

```text
User command
  -> cheap router
  -> role selector
  -> native read-only subagent fan-out
  -> bounded independent reports
  -> synthesis judge
  -> chat answer or action contract
```

### Default Role Pool

- `architect`: design, coupling, boundaries, long-term structure.
- `skeptic`: hidden assumptions, failure modes, regressions, security/product risks.
- `implementation_analyst`: concrete implementation path, files likely touched, sequencing.
- `tester`: validation, regression proof, observability, missing tests.
- `maintainer`: repo conventions, docs, existing patterns, future-agent readability.
- `domain_scout`: optional role for unfamiliar domain or research-heavy questions.

### Router Heuristics

- Short/simple task: two roles, usually `implementation_analyst` + `skeptic`.
- Design/refactor/planning task: `architect`, `skeptic`, `maintainer`.
- Bug/debug task: `implementation_analyst`, `tester`, `skeptic`.
- Test/validation task: `tester`, `implementation_analyst`, `maintainer`.
- Docs/process task: `maintainer`, `skeptic`, `architect`.
- Explicit `--roles` overrides heuristics.

### Report Schema

Each role should return a bounded report:

```yaml
role:
verdict:
confidence:
key_findings:
evidence:
risks:
recommended_actions:
open_questions:
```

The synthesizer should return:

```yaml
recommendation:
why:
dissent:
risks:
next_actions:
confidence:
action_contract:
```
