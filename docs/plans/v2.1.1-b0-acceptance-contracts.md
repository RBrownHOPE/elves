# Plan: v2.1.1 B0 and Acceptance Contract Consistency

## Mission

Ship a patch release that makes Batch 0 a supported staging batch everywhere, treats bare and
bracketed stable-ID rows as equivalent, and moves syntax and plan/session drift failures ahead of
worker launch for both Claude Code and Codex hosts.

## Product decisions

- `B0` and legacy numeric `0` are valid batch identities. Ambiguous leading-zero aliases such as
  `B00`, `00`, and `Batch 00` are invalid.
- `- [ ] B0-A1: criterion` and `- [ ] [B0-A1] criterion` are equal spellings. Neither is preferred.
- Stable-looking malformed definition rows receive a line-specific error showing both valid
  replacements. Narrative references to an ID are not definitions and remain ordinary prose.
- The plan is authoritative for criterion text. Session rows can be generated explicitly from it,
  and preflight blocks missing, duplicate, unrelated, or text-mismatched rows before launch.
- Trusted production full-run prepare requires a canonical session and binds its plan/session
  contract to the exact worker packet. Launch and reconciliation fail closed for missing or changed
  bindings, including production state prepared before this patch.
- Final evidence remains a separate landing concern; staging validation does not claim that open
  checkboxes or completion proof have passed.

## Intentional public-contract changes

The following compatibility-gate surfaces change intentionally for 2.1.1:

- `cobbler_agents implement gate`, `launch`, `resume-batch`, and `rollback-ref` accept Batch 0 in
  every batch selector while continuing to reject negatives and ambiguous leading zeros.
- `references/implement-done-report.schema.json` lowers the minimum numeric batch from 1 to 0.

These are the requested B0 compatibility expansion. They are release-scoped in
`api-break-approvals.json`; unrelated public surfaces remain blocking.

- Full `python3 scripts/verify_repo.py --ci --version 2.1.1 --base-ref origin/main` release gate.
