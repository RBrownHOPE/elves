# Plan: Elves Math Research Module

## Mission

Add a configurable math-research module to Elves as a lightweight beta workflow kit. The module
should support preliminary mathematical exploration, proof search, source audit, paper drafting,
and post-draft review without requiring a custom service or one fixed model stack.

Done means a user can start with a rough mathematical goal, run independent subfield scouts, rank
plausible quick wins, and carry promising claims through proof, source, model-review, and
human-verification ledgers.

## Scope

### In Scope

- Math-specific workflow references, templates, prompts, and provider configuration examples.
- OpenRouter-first provider guidance with optional native Gemini, Claude, xAI, OpenAI, and Exa
  hooks.
- A Discovery Sprint pattern for rough goals where the possible theorem is not known yet.
- Consistency checks so the math-module guidance stays aligned across the Elves docs.

### Out of Scope

- A new CLI, daemon, database, or hosted service.
- Hardcoded model names or mandatory paid tools beyond the OpenRouter baseline.
- A formal-proof assistant integration beyond optional scout guidance.
- Project-specific private logs, keys, or unpublished geometry-project artifacts as defaults.

## Non-Negotiables

- Treat model output as research assistance, not mathematical authority.
- Keep v1 portable and documentation-driven.
- Use OpenRouter as the minimum expected model provider; keep all native providers optional.
- Preserve the stage-then-launch and PR-based Elves workflow.

## Test Strategy

- **Primary gate:** `python3 scripts/check_repo_consistency.py`
- **Install health:** `python3 scripts/install_doctor.py --doctor`
- **Installed-skill drift:** `python3 scripts/sync_installed_skills.py --check`
- **Manual review:** inspect new math references for provider configurability, source/proof
  discipline, and no overclaiming about model authority.

## Notes

- Preliminary research is first-class. The workflow should not assume the theorem is already known.
- Discovery scouts should include relevant and adjacent subfields. Useful defaults include
  geometry/topology, PDE/spectral theory, convexity/optimization, probability/analysis,
  algebraic/combinatorial analogs, numerical experimentation, and formalization prospects.
- The synthesis step should rank opportunities by tractability, novelty, verification burden, and
  likely value to a human mathematician.
