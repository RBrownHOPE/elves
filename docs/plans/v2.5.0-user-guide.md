# Plan: Elves user guide and GitHub Pages

## Mission

Give new and returning users one clear, task-first guide to installing Elves, starting a run,
choosing and watching a worker, reviewing the result, and deciding whether to merge. Publish the
guide from this repository through GitHub Pages, link it from the README, and release the current
adaptive worker work with the guide as Elves v2.5.0.

## Planning classification

- **Execution reasoning:** `low`. This is static HTML, documentation alignment, and a standard
  GitHub Pages workflow.
- **Review risk:** `standard`. Incorrect commands or host-specific wording could confuse users, and
  the deployed page needs responsive and accessible checks.
- **Review focus:** task order, exact commands, Claude Code and Codex parity, Pages deployment, and
  restrained writing and design.

## Scope

### In scope

- A standalone static guide under `guide/` with no build dependency.
- A same-repository GitHub Actions workflow that publishes `guide/` to GitHub Pages.
- README, changelog, end-to-end reference, durable docs, and release metadata updates.
- A narrow native-worker status-race fix if release CI exposes the prior adaptive-worker defect.
- A critical Gemini 3.5 clarity review, browser checks, focused repository verification, landing by
  regular merge commit, a v2.5.0 GitHub release, and refreshed local Claude Code and Codex installs.

### Out of scope

- New orchestration behavior or runtime worker routes.
- A separate Pages repository, custom domain, analytics, or marketing site.
- Changes to merge authority or worker authority.

## Batch 0 [B0]: Build and publish the user guide

**Coordinator-to-implementer handoff:**

- **Intent / why:** make the current Elves flow understandable without reading the full workflow
  contract.
- **Non-obvious rationale:** this is a usage guide, not marketing. Lead with tasks, use exact prompts
  and commands, and disclose capability-dependent fallbacks.
- **Build On targets:** `README.md`, `references/adaptive-worker-routing.md`,
  `references/e2e-chat-to-land.md`, the existing Elves colors, and current GitHub Pages actions.
- **Owned surfaces:** `guide/`, Pages workflow, public docs, durable docs, release metadata, and
  focused documentation checks.
- **Forbidden surfaces:** runtime behavior, provider credentials, protected refs, and unrelated
  plans or archived run records.
- **Acceptance evidence:** semantic and link checks, responsive browser views, humanizer scan,
  Gemini 3.5 review and revisions, repository verification, GitHub checks, and live Pages response.
- **Failure modes / pitfalls:** do not copy stale worker wording, imply cache transfer, imply worker
  merge authority, add a build dependency, or turn the page into promotional copy.
- **HEAD / run-doc paths / route-session identity / output format:** build from the staged branch
  tip and use the run paths recorded in `.elves-session.json`; the host owns commits, PR, review,
  merge, release, and installed-skill refresh.

**Constitution impacts:** host parity, honest capability reporting, user-owned merge authority.
**Review focus:** completeness, exact commands, parity, clarity, accessibility, Pages scope.
**Focused tests:** HTML parser/link checks, local HTTP/browser checks, humanizer scan,

**Depends on:** none.

## Master acceptance

- [x] [M-A1] The self-contained Pages artifact gives a user one complete, current Elves workflow without requiring internal contracts first.
- [x] [M-A2] The guide, README, changelog, host references, and durable docs use accurate Claude Code and Codex invocation wording and preserve the same authority model.
- [x] [M-A3] The reviewed documentation PR contains aligned v2.5.0 release metadata and is ready for host-owned merge, tag, Pages enablement, and global install refresh.

## Non-negotiables

- Keep the guide practical and compact. It is documentation, not marketing.
- Keep Claude Code and Codex workflow semantics equivalent while using honest host-specific syntax.
- Do not give a worker PR, protected-ref, or merge authority.
- Use a regular merge commit only after final readiness.
- Do not use em dashes or emojis in new or revised public writing.

## Test strategy

- Parse the HTML and verify local anchors and external links.
- Inspect desktop and narrow browser renders, keyboard focus, and responsive overflow.
- Run a targeted humanizer phrase scan over every changed prose surface.
- Run documentation consistency checks and the final repository verification once after revisions.
- After merge, verify the Pages deployment and both installed skill bundles.

## Post-landing host actions

These actions happen after the landability gate because they require the merge commit. They are not
worker or pre-merge acceptance criteria.

- Enable GitHub Pages with workflow publishing, wait for the deployment, and verify
  `https://aigorahub.github.io/elves/`.
- Tag the merge as v2.5.0 and publish the GitHub release.
- Sync both global installs from the merged `main` tip and run their installed-bundle checks.
