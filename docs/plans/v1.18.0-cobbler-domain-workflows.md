# v1.18.0 Cobbler Domain Workflows

## Goal

Make Elves read as one coherent system:

- Elves is the execution system.
- Cobbler is the default coordinator and router.
- Domain workflows are specialized packs Cobbler can route into.
- Math is the first domain workflow.
- Provider-backed model diversity is optional evidence, not a default dependency.

This release should keep all existing Cobbler, Council compatibility, model-routing, and math
workflow functionality while removing the impression that math is a peer orchestrator or that
OpenRouter is required for ordinary use.

## Batches

### Batch 1: Architecture and Session State

Update the main skill docs, repo agent instructions, README, Cobbler docs, and templates so an
Elves invocation activates Cobbler-first coordination for the current Elves session. Real Elves
runs persist that state in the survival guide and `.elves-session.json`; one-off Cobbler remains
read-only and stateless.

Acceptance:

- Runtime docs explain the hierarchy: Elves -> Cobbler -> domain workflows -> providers.
- Survival guide template has a Cobbler Session State block.
- `.elves-session.json` guidance includes a `cobbler.default_for_session` shape.
- Cobbler Mode remains current-thread chat state unless recorded by a real Elves run.

### Batch 2: Math as a Cobbler Domain Workflow

Reframe the math workflow as a Cobbler-managed domain workflow. Preserve Discovery Sprint, scout
lanes, claim lifecycle, source audit, adversarial review, ledgers, provider role slots, and human
verification.

Acceptance:

- Math docs say Cobbler coordinates the math workflow.
- OpenRouter is recommended for broad math model diversity, not required by default.
- Missing provider keys fall back to native subagents or direct analysis with confidence noted.
- Config examples no longer make `OPENROUTER_API_KEY` a required default for math.

### Batch 3: Guardrails, Tests, and Release Readiness

Add consistency checks and tests that prevent drift back to Council-primary, math-as-peer, or
provider-required wording. Update changelog and final docs, run validation, get independent review,
land the implementation PR, then cut the release bump and tag.

Acceptance:

- Repo consistency checks cover Cobbler session state and domain workflow hierarchy.
- Unit tests cover new drift checks.
- `python3 scripts/check_repo_consistency.py` passes.
- `python3 -m unittest discover` passes.
- Implementation PR lands with a merge commit.
- Post-merge release bump tags GitHub as `v1.18.0`.

## Non-Goals

- Do not remove Council aliases.
- Do not remove provider-backed council.
- Do not remove OpenRouter math examples.
- Do not create a new runtime, daemon, or provider abstraction.
- Do not make Quick Cobbler stateful or repo-mutating.

## Review Focus

- Does every user-facing doc make Cobbler the coordinator rather than an optional add-on?
- Does math retain its useful workflow details after the reframing?
- Is provider wording native-first and optional without underselling configured external routes?
- Do the consistency checks catch the specific drift this release is fixing?
