# Elves 2.3: Joyful Runs Rewrite

## Goal

Make Elves fast, calm, and satisfying to use: plan clearly, delegate confidently, review
intelligently, and ship. Rewrite the organically grown prose and control plane from a clean design
while preserving the proven security and runtime kernel.

The normal trusted-worker path is one risk-aware plan, one autonomous worker goal, meaningful
worker commits and pushes, a parked driver, a visible non-model progress stream, one cumulative
terminal review, consolidated fixes, delta-only re-review, impact-selected proof, and one of two
host-owned outcomes: a landable PR or an explicitly authorized regular merge commit.

## Run posture

- `risk`: `low | standard | high`; `trust_mode`: `trusted | untrusted` is separate.
- Work driver: trusted Grok Build full run on the assigned feature branch.
- Driver posture: parked during healthy execution; no routine mid-run review or timed chat updates.
- Landing outcome: landable PR. The worker never merges; the driver is not authorized to merge in
  this run unless the user later explicitly changes Run Control.
- No planned mid-run model checkpoint. Deterministic safety tripwires and material scope/assumption
  changes may wake the driver.

## Safety kernel to preserve

- Exact plan/session/packet acceptance identity, including B0/B1 and bare/bracketed stable IDs.
- Credential, origin, branch, worktree, ancestry, clean-tip, protected-ref, and redaction guards.
- Worker merge/tag/protected-ref/PR/landing-policy authority is always false.
- Test integrity, constitution compliance, exact-HEAD readiness, independent terminal review, and
  required final CI.
- Strict detached/import evidence for untrusted writers.
- Native Claude Code and Codex operation remains valid without Grok or optional providers.

## Architecture

```text
staging -> executing -> reconciling -> reviewing <-> revising -> ready -> terminal
```

Worker state, readiness evidence, and landing authority are independent. `ready=true` never grants
merge permission; `driver_authorized=true` never proves readiness. Merge requires both at the same
exact HEAD.

## Batch 0 [B0]: Canonical contract and migration ledger

**Risk:** high

**Caution:** A fresh rewrite can silently omit safeguards. Inventory every normative requirement
before deleting old prose; keep the ledger temporary and proof-oriented rather than turning it into
another manual.

**Affected surfaces:** canonical workflow documents, templates, consistency policy, migration
ledger.

**Constitution impacts:** preserve model-neutral operation, user-owned merge authority, trusted and
untrusted safety boundaries, and unattended recovery.

**Review focus:** omissions, conflicting authority, accidental retention of old ceremony, and
duplicate sources of truth.

**Focused tests:** acceptance parsing and semantic consistency fixtures affected by the new contract.

**Acceptance criteria:**

- [ ] B0-A1: Normal flow, actor authority, state transitions, wake conditions, proof rules, and terminal outcomes are defined once.
- [ ] [B0-A2] Every retained safety-kernel requirement has a canonical destination and a proving test.
- [ ] B0-A3: Risk is low, standard, or high while trust mode is independently trusted or untrusted.
- [ ] [B0-A4] Claude Code and Codex share identical workflow semantics.

## Batch 1 [B1]: Host-owned authority and exact-HEAD readiness

**Risk:** high

**Depends on:** B0

**Caution:** Treat landing authorization as immutable host control. Do not infer it from worker
reports, generic completion language, or readiness itself.

**Affected surfaces:** session schema, report reconciliation, behavior policy, landing checker or a
small host-only landing guard, focused authority tests.

**Constitution impacts:** no worker merge authority; no landing before full completion and review;
regular merge commits only when explicitly authorized.

**Review focus:** hostile worker fields, stale readiness, changed HEAD, active-run `/land-pr`, and
chat-to-work/chat-to-land parity.

**Focused tests:** full-run reconciliation, storage isolation, landing checker, behavior policy.

**Acceptance criteria:**

- [ ] B1-A1: Worker evidence cannot grant merge authority or modify the host-owned landing outcome.
- [ ] [B1-A2] Complete-without-merge and complete-and-merge share one implementation and readiness pipeline.
- [ ] B1-A3: Active-run land-pr grants driver authority without bypassing or restarting readiness.
- [ ] [B1-A4] Readiness is attested to an exact HEAD and changed inputs invalidate only affected proof.
- [ ] B1-A5: A host-only merge guard requires completed acceptance, resolved blockers, clean exact-tip review evidence, required checks, and explicit driver authorization.

## Batch 2 [B2]: Default live worker window and deterministic sentinel

**Risk:** high

**Depends on:** B0

**Caution:** Keep shared-OAuth transcript restrictions and redaction intact. The visible stream is
observable activity, not hidden reasoning and not a new model-based watcher.

**Affected surfaces:** full-run CLI, supervisor events, await/monitor/log output, worker packet,
recovery tests, operator docs.

**Constitution impacts:** quiet parked driver, visible progress, prompt recovery from worker death,
and credentials never exposed through telemetry.

**Review focus:** process death, hangs, false completion, event corruption, credential leakage,
branch progress preservation, and no driver model calls while following.

**Focused tests:** full-run supervisor, CLI lifecycle, redaction, credential isolation, worker death
and stale-heartbeat fixtures.

**Acceptance criteria:**

- [ ] B2-A1: Trusted full-run await follows a sanitized human-readable worker stream by default with a quiet opt-out.
- [ ] [B2-A2] Follow mode performs no model inference and replaces timed driver chat updates.
- [ ] B2-A3: Death, hangs, missing or malformed completion, safety violations, explicit blockers, and material scope or assumption changes wake the driver.
- [ ] [B2-A4] Previously pushed trusted worker progress survives recovery and can resume safely from the verified feature tip.
- [ ] B2-A5: Shared OAuth raw-transcript restrictions and all redaction guarantees remain intact.
- [ ] [B2-A6] Workers commit and push meaningful progress slices with concrete operator-readable subjects.
- [ ] B2-A7: Native Grok goal mode is used only when capability-proven; otherwise the compatible one-packet fallback is recorded honestly.

## Batch 3 [B3]: Risk-directed proof and convergent review

**Risk:** high

**Depends on:** B0, B1

**Caution:** Evidence reuse must be input-based and conservative around shared, security, runtime,
dependency, migration, and public-interface changes. Faster must not mean green-seeking.

**Affected surfaces:** risk policy, evidence review, verification helpers, reviewer protocol,
cleanup attestation, tests.

**Constitution impacts:** sufficient current-tip proof, one strong terminal review, no infinite
review/revise loop, and test integrity.

**Review focus:** invalidation scope, high-risk escalation, cleanup-only reuse, admission of new
blocking findings, and final stopping rule.

**Focused tests:** risk/evidence policy, verify_repo, reviewer scenario fixtures, cleanup proof.

**Acceptance criteria:**

- [ ] B3-A1: Verification follows an explicit changed-surface to affected-consumer to selected-test impact path.
- [ ] [B3-A2] Evidence records its inputs and invalidation scope so unchanged proof is reused.
- [ ] B3-A3: One cumulative review checks completeness, constitution, declared risks, and concrete regressions.
- [ ] [B3-A4] Blocking findings are consolidated before revision and advisory findings do not delay readiness.
- [ ] B3-A5: Re-review verifies the revision delta and unresolved blockers without rescanning settled untouched work.
- [ ] [B3-A6] New re-review blockers require a concrete serious regression, acceptance or constitution breach, security or data-integrity issue, or failure introduced by the revision.
- [ ] B3-A7: Cleanup-only operational changes do not invalidate product proof, while relevant new surfaces broaden proof automatically.
- [ ] [B3-A8] The loop stops on sufficient exact-tip evidence rather than the absence of reviewer suggestions.

## Validation Budget

- During implementation, run affected test modules and deterministic checks for changed surfaces.
- Do not run the full suite after every batch, commit, or review fix.
- Re-review only the delta from the last reviewed SHA and rerun only invalidated tests.
- Run one broad local suite on the final unchanged runtime/test tree, then use required GitHub CI for
  final platform breadth.

## Explicitly Out of Scope

- A continuously reasoning watcher agent.
- A default multi-Grok parallel-lane scheduler or distributed lock system.
- Exposure of hidden model reasoning or unsafe shared-OAuth raw transcripts.
- External provider requirements.
- Per-batch driver review for a healthy trusted full run.
- Broad proof reruns after every revision.

## Trusted Worker Contract

Implement this entire plan in one persistent run. Use your judgment: the batches describe outcomes,
risk, and proof, not an editing script. Reorder independent work when useful, make reasonable
adjacent changes, and extend existing modules rather than building parallel machinery.

### Authority and owned surfaces

- You may edit product code, tests, `SKILL.md`, `AGENTS.md`, README, CHANGELOG, references,
  templates, config examples, release metadata, and durable architecture docs needed by this plan.
- Do not edit `docs/elves/survival-guide-joyful-runs.md`,
  `docs/elves/execution-log-joyful-runs.md`, `.elves-session.json`, `.elves/runtime`, credentials,
  other worktrees, protected refs, PR state, or landing policy.
- Commit and push meaningful independently reviewable progress on `codex/joyful-elves-runs` with
  concrete subjects such as `[codex/joyful-elves-runs · Batch N/6 · Implement] <outcome>`.
- Never merge, tag, open another PR, alter protected refs, or treat your own review as the required
  independent terminal review.
- Do not pause for routine implementation choices. Stop only for a genuine blocker, safety
  conflict, material scope/assumption change, or inability to preserve an acceptance criterion.

### Verification and closeout

- Run focused tests for changed and affected surfaces while implementing. Run the full local suite
  once on the final unchanged runtime/test tree, not after every batch or fix.
- Update human-facing docs and version metadata to 2.3.0 as part of the plan.
- Finish with a clean worktree and no unpushed commits.
- Every acceptance row in this document must appear verbatim by ID and criterion in the final
  report with `met: true` and concrete evidence.

### Full-run event contract v1

Use the environment paths `ELVES_FULL_RUN_EVENTS` and `ELVES_FULL_RUN_REPORT`. Preserve the exact
`ELVES_FULL_RUN_RUN_ID`, `ELVES_FULL_RUN_ATTEMPT`, session ID, branch, and launch HEAD from the
pre-created baseline report. Append events to the JSONL file; never truncate it. Atomically replace
the report after material state changes.

Each event requires `timestamp` (UTC ISO-8601), `session_id`, `branch`, `head` (exact commit SHA),
`batch` (integer; 0 for setup), `type`, and a redacted `summary` of at most 500 characters. Allowed
types are `run_started`, `heartbeat`, `batch_started`, `commit_pushed`, `gate_result`,
`batch_complete`, `high_risk_checkpoint`, `blocked`, and `run_complete`. `checkpoint_id` is required
only for a staged `high_risk_checkpoint` and forbidden otherwise. Emit at most one terminal event:
`blocked` or `run_complete`.

The final report is one JSON object requiring `run_id`, `attempt`, `session_id`, `branch`,
`start_head`, `final_head`, `status`, `batches`, `acceptance`, and `commits`. A complete report uses
`status: "complete"`; has one non-empty `{id, status: "complete", evidence}` batch record for every
batch; has the exact full set of `{id, criterion, met: true, evidence}` acceptance rows staged from
this document; lists the exact ordered `start_head..final_head` commit chain as SHA strings or
`{sha, subject}` objects; and has empty `blockers` and `remaining_risks` arrays. Optional
`docs_changed` is an array. A terminal event is only a wake signal; the report and host audit remain
completion authority.
