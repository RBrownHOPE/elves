# Plan: v2.1.0 Delegated Worker and Runtime Stabilization

## Mission

Ship Elves v2.1.0 with a real delegated full-run mode: after a Claude Code or Codex driver stages a
trusted Grok Build run, one persistent worker owns feature-branch implementation, tests, commits,
pushes, and structured progress while the driver parks. The same release also closes the confirmed
packaging, runtime correctness, isolation, evidence, CI, and documentation defects found during the
repository review, then proves both host installations and lands the reviewed PR with a regular
merge commit.

## Product Decisions

- Host-native remains the zero-setup default. Grok Build is an explicit optional work driver.
- A trusted Grok `fast` lane defaults to `delegation_scope=full_run`,
  `git_mode=branch_progress`, and `driver_monitor_mode=event_driven`.
- The worker receives one self-contained packet for all batches and keeps one exact session. It does
  not wait for a driver prompt between batches.
- The worker may edit, test, commit, and push only the staged feature branch. The driver retains
  plan/run-control memory, PR authority, final independent review, and merge authority.
- The raw worker transcript stays in a private local artifact. Driver chat/status consumes only
  bounded events: active batch, branch head, checks, heartbeat, blockers, and next action.
- The parked driver wakes only for a blocker/error, stale heartbeat, safety tripwire, explicitly
  planned high-risk checkpoint, user input, worker exit, or final readiness.
- Claude Code and Codex share the same state machine and safety semantics. Only host-native
  invocation and continuation plumbing differ.
- Test updates are allowed when behavior legitimately changes and coverage is preserved or improved;
  weakening or deleting tests merely to obtain green remains forbidden.

## Scope

### In Scope

- Delegated full-run supervision, bounded telemetry, branch authority, and host wake conditions.
- Structured behavior policy for direct edits, bounded tasks, full runs, kickoff, landing, and
  trusted versus untrusted worker routing.
- Self-contained Claude Code and Codex installed bundles and real installed-artifact smoke tests.
- Built-in adapter identity preservation for Claude, Codex, Grok, Gemini, Antigravity, and OpenCode.
- Exact session identity, atomic lease storage, durable audit evidence, cumulative patch checks, and
  one-to-one landing evidence.
- Tracked-source isolation for external read-only lanes, minimal execution environments, redaction,
  notification safety, and temporary-artifact cleanup.
- Run-scoped rollback refs, safe delegated branch reconciliation, cached preflight evidence,
  evidence-aware review, and an implemented public-API surface snapshot helper.
- Decomposition of the large dispatch attempt path and separation of consistency policy data from
  the generic checker, backed by characterization tests.
- README information architecture, every current reference/alias/AI doc, v2.1.0 metadata, and the
  changelog.
- Full local, installed-bundle, live-host, PR-feedback, CI, and final landing proof.

### Out of Scope

- Making Grok Build, OpenRouter, or any external provider mandatory for ordinary Elves use.
- Supporting OpenCode or Antigravity as the main Elves orchestrator.
- Publishing a GitHub release, package, or tag; the versioned code and changelog may land without a
  publication action.
- Changing credentials, rotating user secrets, or modifying installed user skills during the run.
- Squash/rebase landing or force-pushing any branch.

## Worker Contract Shared by Every Batch

- **Intent / why:** eliminate nested-driver orchestration tax while making optional external routes
  safer, more deterministic, and honestly documented.
- **Non-obvious rationale:** full-run delegation is a first-class trusted branch-progress path, not
  the detached untrusted writer lease. Security hardening must not collapse these two modes.
- **Build On:** current `cobbler_runtime` registries and state stores, existing Lane A implement CLI,
  workspace guard, landing checker, install synchronizer, and repo consistency tests.
- **Owned surfaces:** tracked product code, tests, workflows, public docs, references, aliases,
  examples, and durable AI docs named by the batches below.
- **Forbidden surfaces:** `.git` internals, base branch, merge/tag operations, credentials, ignored
  local model config, canonical run-control fields in the Survival Guide and `.elves-session.json`,
  other worktrees, user-installed skills, and local application state.
- **Acceptance evidence:** per-batch focused tests plus the final canonical gate; exact command,
  exit status, branch head, and criterion-specific evidence in the full-run report.
- **Failure modes / pitfalls:** do not use the existing untrusted lease path as security proof before
  it is repaired; do not launch subprocesses with the inherited host environment; do not paper over
  failing tests or phrase-check contradictions; preserve Python standard-library-only operation.
- **HEAD / run docs:** start at `48e6fc58c763a58d8eb38090bcd94b9aeebfe768` on
  `codex/delegated-worker-v2-1`; plan, survival guide, and execution log are the paths recorded below.
  The packet supplies the exact Grok session ID and requires a machine-readable final report.

- [ ] Require observed adapter, model, sandbox, worktree, CWD, parent, source HEAD, capabilities, and

- [ ] Put malformed-record scanning, exclusivity checking, and lease creation under one lock; use

**Acceptance criteria:**

- [ ] B3-A1 — Colliding and traversal-shaped IDs cannot collide, escape runtime storage, or retrieve

- [ ] B3-A2 — Concurrent session/lease operations leave valid JSON and allow exactly one live lease;

- [ ] B3-A3 — Read-only commands do not create runtime directories; arbitrary/unregistered sessions

- [ ] B3-A4 — Missing, mismatched, stale, preference-declared, or unsupported qualification evidence

## Non-Negotiables

- The driver owns run control, PR state, final independent review, and merge; Grok never merges,
  tags, changes the base branch, force-pushes, or opens another PR.
- No credential value enters a packet, event, transcript summary, commit, test fixture, PR surface,
  notification, or final response.
- Preserve native-only operation and Python standard-library-only runtime; optional providers never
  become a base requirement.
- Fix root causes and preserve or improve test coverage. Legitimate behavior-driven test changes are
  allowed and must carry explicit evidence; weakening tests merely to turn a gate green is forbidden.
- Landing is a regular merge commit only after exact acceptance evidence, fresh cumulative review,
  all PR feedback, and required checks are green.

## Test Strategy

- **Focused:** affected `unittest` modules and fixture integration tests after each implementation
  slice.
- **Canonical local gate:** the new single repository verification command added in Batch 1.
- **Release gate:** `python3 scripts/check_repo_consistency.py`,
  `python3 scripts/release_checklist.py --version 2.1.0`, full unittest discovery, recursive
  `compileall`, shell syntax, JSON validation, `git diff --check`, installed-bundle smokes, API
  compatibility, secret scan, landing check, and link resolution.
- **Live canaries:** bounded Claude Code, Codex, and real Grok Build checks without altering installed
  user state.
- **Known flaky tests:** none. A discovered flaky test must be diagnosed, not silently ignored.

## Notes

- This is a finite chat-to-land run explicitly authorized by the user. The terminal condition is a
  safely merged PR or a genuine blocker that cannot be worked around without new authority.
- The existing fast Lane A, not the currently suspect detached writer lease, bootstraps this run.
  Launch it with an explicitly minimized environment and private redirected transcript until the
  new supervisor/redaction boundary exists.
- The driver will not duplicate the worker's batch loop. It stages once, parks, then performs one
  final cumulative readiness and landing loop.
