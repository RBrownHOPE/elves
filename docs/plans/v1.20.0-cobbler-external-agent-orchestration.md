# Plan: v1.20.0 Cobbler External-Agent Orchestration

## Mission

Turn Elves' existing advisory model-routing language into a small, real, configurable external-agent
harness inside Cobbler. A user should be able to plan with independent agents in parallel, assign one
qualified agent one substantial implementation batch, have an independent review council inspect the
result, and repeat—while the host coordinator remains accountable for canonical run documents, branch
refs/commits, pushes, PRs, validation, synthesis, and fallback decisions. Strong hosts may delegate
bounded routine checks to a configured cheaper reviewer without changing the supervising model.

The default experience must still work with no external tools or provider keys. Users who have Claude
Code but not Grok Build, Sakana but not Claude, an API-only model through OpenRouter, a future Google
or Antigravity tool, or a private wrapper should be able to map those capabilities to roles without
changing Elves source code or accepting false claims about what their tools can do.

Done means the repository contains a tested standard-library runtime, documented configuration and
setup flow, safe persistent-session and external-writer contracts, matched Claude/Codex skill surfaces,
a `/setup-council` compatibility helper, and a master CouncilElves launch prompt. The current user's
qualified Fable 5, Grok 4.5, and Fugu Ultra routes must pass a final live end-to-end experiment, but
those model names must remain a project preset—not the public default.

## Why this plan is unusually explicit

The main coordinator is expected to be the strongest agent in the run. It writes the plan, Survival
Guide, batch contracts, execution log, learnings, and context packets for an implementation worker
that may be less capable, less careful, or missing the conversation context. Therefore every handoff
below includes intent, non-obvious rationale, existing surfaces to extend, exact ownership boundaries,
forbidden shortcuts, likely failure modes, and acceptance evidence.

This is not permission to prescribe brittle line-by-line code. The implementer should still survey the
current checkout and use judgment. It is a requirement that the worker can execute the batch correctly
without relying on this chat, model prestige, or unstated architectural knowledge.

## Current architecture to preserve

Elves already has the right hierarchy and most of the policy language:

1. **Elves executes** plans through branches, PRs, validation, review, memory, and landing.
2. **Cobbler coordinates** intent, capabilities, routes, context, evidence, dissent, output medium,
   and one fitted answer.
3. **Domain workflows specialize** Cobbler for a type of work.
4. **Providers and harnesses are optional routes**, not orchestration layers.

Existing surfaces to extend rather than duplicate:

- `SKILL.md` and `AGENTS.md`: canonical Claude-compatible and Codex behavior mirrors.
- `references/council-workflow.md` and `references/council-prompts.md`: context-packet, independent
  lens, worker-scope, evidence, and fitted-answer contracts.
- `config.json.example`, `references/tool-config-examples.md`, and the Survival Guide model-routing
  block: native-first preferences and explicit `required` semantics.
- `scripts/workspace_guard.py`: git-command classification and owned-tip safety concepts.
- `scripts/preflight_worktree.py`: deterministic worktree creation and collision checks.
- `scripts/elves_landing_check.py`: acceptance-evidence and final-readiness guardrail.
- `scripts/sync_installed_skills.py`: managed Claude/Codex runtime and alias installation.
- `scripts/check_repo_consistency.py`: cross-file behavioral phrase pins.
- Existing `unittest` fake-repository/fake-executable patterns under `tests/`.

Do not create a second orchestration hierarchy, a separate Council ledger, a provider-key store, or
an external-agent-owned branch/remote workflow.

### Compatibility facts that are design inputs

- Grok Build `0.2.93` silently ignored headless `--worktree --resume` and retained the source CWD.
  That combination must be recorded as behaviorally broken; it must never count as isolation.
- Interactive `grok -w -r <parent-id>` created a detached worktree by automatically forking the
  complete parent transcript into a discoverable child ID. Exact UUID plus changed CWD is not
  supported; exact persistent **lineage** is.
- Resuming that discovered child ID from its recorded worktree worked headlessly and preserved the
  implementation context.
- A narrow Grok `--tools` filter can remove hidden support tools and make the terminal bundle invalid.
  Capability profiles are complete, versioned bundles—not independently composable booleans.
- Grok documents the CLI `dontAsk` value as accepted but not fully wired. Do not treat it as a hard
  boundary. Explicit deny policy, hard hooks/custom sandbox where available, credential isolation,
  and post-turn audit are required layers.
- Grok child-process network restriction is documented as a no-op on macOS. Credential scrubbing
  and egress policy must be separate capabilities.
- A resumed Grok session cannot change its startup sandbox, even with `--fork-session`. The original
  `workspace` child could edit its CWD but `git add` failed on the shared worktree `index.lock`.
  A separately created, context-seeded `devbox` successor passed a one-file detached-commit canary.
  Persist the actual sandbox profile as part of the write capability; never infer commit capability
  from edit capability.
- A qualified detached worker commit is not branch ownership. The host still denies branch/tag/push,
  snapshots refs/remotes/config/hooks, audits the complete first-parent commit chain, and imports
  reviewed binary patches into its own branch under host-authored commits.
- Codex model choice is per invocation. A strong supervising thread can launch a bounded, ephemeral
  cheaper model for low-risk read-only review. Deterministic git and PR operations do not need an LLM.
- Fugu may inherit unrelated Codex MCP configurations. Stale MCP OAuth warnings do not mean Sakana
  inference failed; model health and optional MCP health must be reported separately.
- Claude and Fugu return observed token usage; none of these subscription harnesses reliably exposes
  remaining quota. `unknown` is a valid value. Elves must not invent limits.

## Product decisions already made

### 1. The host coordinator is accountable

The active Codex or Claude host owns:

- the plan, Survival Guide, execution log, learnings, and `.elves-session.json`;
- capability resolution, task packets, dissent synthesis, route changes, and fallback decisions;
- the active branch, PR, branch commits, pushes, tags, comments, and final Elves Report;
- validation and acceptance evidence;
- importing or rejecting an external worker's patch.

An external worker may edit only within an issued lease. A write profile may permit meaningful
commits on detached HEAD when that exact capability has passed a canary; those commits are untrusted
handoff boundaries, not branch history. The worker never creates/moves refs, pushes, opens or owns
the PR, edits canonical run memory, chooses its own scope, or declares a batch complete.

### 2. Native-first remains the public default

An ordinary Elves run with no extra CLIs must continue through host-native agents or direct analysis.
Missing Claude Code, Grok Build, Sakana, OpenRouter, or any future provider is a capability result,
not an installation failure. Optional routes fall through their configured chain and record a reason
only when the change affects risk or confidence.

`required = true` is valid only when the project Survival Guide explicitly opts a role or phase into
a hard dependency. This run explicitly requires the exact persistent Grok implementation successor
for product implementation and remediation across all six batches; Claude/host alternatives require
a new user decision rather than automatic fallback. That project override does not change the
shipped default.

### 3. Configuration describes roles and capabilities, not prestige

The public configuration maps role slots and fallback chains to named harness profiles. It does not
hardcode “best” models in orchestration code. Recommendations should prefer capability language such
as “strongest qualified coding agent with command and isolated-write support” or “independent model
family for review.” Dated examples may name current models, but setup must probe what is actually
installed and available.

Local project preferences live in the intentionally gitignored `.elves/models.toml`; setup writes
that file for the current checkout and must never stage or commit it. The tracked
`references/models.toml.example` defines the schema without machine paths or credentials, while the
active Survival Guide commits the resolved route snapshot needed to reproduce and review a run.
Existing installed/user `config.json` may provide machine defaults; `config.json.example` remains a
shipped example and compatibility surface, not another setup-written project store. Active-run
precedence is:

1. explicit Survival Guide route and `required` values;
2. project `.elves/models.toml`;
3. installed/user `config.json` defaults;
4. native host defaults.

No configuration file stores raw credentials. It may name environment variables or local credential
stores. Generated capability and session state belongs under ignored `.elves/runtime/`.

| Surface | Owner and lifetime | Committed? | Purpose |
| --- | --- | --- | --- |
| `config.json` | installed/user machine default | no in this repo | backward-compatible defaults |
| `.elves/models.toml` | local checkout preference | no; intentionally ignored | role choices and fallback chains for future runs |
| `references/models.toml.example` | Elves distribution | yes | schema, comments, and safe examples |
| active Survival Guide | host coordinator for one run | yes | resolved routes, explicit `required` policy, and provenance |

Elves does not silently invent a source-controlled team-preferences file. Teams that want a shared
preset can copy a reviewed example through their own project policy, while the run still records the
effective choice in its Survival Guide.

### 4. Harnesses advertise and prove capabilities

A harness profile needs a stable identity plus behaviorally verified capabilities. The initial
contract should cover at least:

- availability, version, authentication state, and model discovery;
- read-only repo inspection, local command execution, and file editing;
- structured output and actual-model reporting;
- persistent session creation/resume and parent/child lineage;
- parallel-safe read-only use;
- isolated-write and worktree behavior;
- permission policy, credential sharing, and network-egress posture;
- usage fields and whether remaining quota is known.

An advertised CLI flag is not proof. A capability is qualified only after its adapter's probe or
fixture demonstrates the behavior. Unknown or unqualified capabilities fail closed for write roles
and fall back for optional read-only roles.

### 5. Council fan-out is parallel and independent

Planning and review lanes launch concurrently when they do not depend on each other. Every lane gets
the same redacted context packet and cannot read peer reports before synthesis. One failed optional
lane does not cancel successful lanes. The coordinator collects bounded reports and returns one fitted
answer with the strongest dissent visible.

The implementation agent is not an independent reviewer of its own work. Review defaults to a fresh
host-native lens plus configured contextual reviewers such as Fable or Fugu.

### 6. Persistence is exact, explicit, and recoverable

Never use ambiguous “last” or “continue most recent” selection. Persist harness, requested model,
actual model, session ID, parent ID, CWD/worktree, source HEAD, context digest, timestamps, status,
and observed usage. Expected context changes cause rehydration from canonical disk state; unexpected
model/CWD/lineage drift blocks a write role.

### 7. External implementation is an untrusted detached-commit and patch workflow

One external writer lease exists at a time. Before a turn, record the worker worktree, clean status,
detached HEAD, full ref snapshot/digest, allowed paths, forbidden paths, sandbox profile, and session
identity. Give the worker one substantial batch per turn. When the qualified profile permits it,
target two to five meaningful direct-descendant commits across that batch; do not force artificial
micro-commits. Stream or summarize progress to chat while the turn runs, but do not import partial
state until the process exits and the lease can be audited. After the turn, verify:

- HEAD moved only through the expected non-merge detached commit chain rooted at the recorded base;
- every worker commit has an auditable parent, tree, changed-path set, author, and outcome-oriented
  subject; status and index are clean at handoff;
- no branch/tag/other ref, remote, shared config, hook, push, or PR mutation occurred;
- every changed/untracked path is in scope and no symlink escapes scope;
- `.git`, `.elves`, run documents, credentials, and other forbidden surfaces are untouched;
- the worker process exited and no paid or long-running child remains ambiguous.

The host exports each approved worker boundary as a binary-safe patch, reviews it as untrusted input,
and applies it to the owned branch only after `git apply --check --index` or equivalent validation.
The host may preserve, combine, or split worker boundaries based on reviewability, then runs tests,
writes run memory, creates sanitized operator-visible commits that record source worker SHAs, and
pushes. Do not use a bare cherry-pick across the trust boundary.

### 8. Git history is an operator progress surface

Users monitor unattended agents through GitHub, GitKraken, and ordinary git history. The host must
therefore commit and push meaningful, reviewable progress slices throughout a batch instead of
keeping hours of completed work only in a working tree or collapsing a batch into one opaque close
commit. Qualified external workers may create detached commits as internal handoff checkpoints; the
host audits/imports their output, validates each accepted slice in proportion to risk, writes any
needed run-memory update, then owns every branch commit and push visible to operators.

Commit subjects use this outcome-oriented form:

```text
[<branch> · Batch <N>/<total> · <phase>] <concrete outcome now true>
```

Phases are a small stable vocabulary such as `Contract`, `Implement`, `Validate`, `Review`, and
`Close`. Do not use vague subjects such as “updates,” “progress,” “WIP,” or “fixes.” A batch may and
usually should have multiple commits when each is independently understandable; do not manufacture
tiny commits that cannot be reviewed. Push every host-owned progress commit promptly, re-read run
state, and reserve `Close` for the commit whose acceptance evidence genuinely closes the batch.

## Proposed runtime shape

Use the existing standard-library Python style. Prefer a thin installed entry point and a focused
package over a new god script:

```text
scripts/cobbler_agents.py                 # user/operator CLI entry point
scripts/cobbler_runtime/
  __init__.py
  schema.py                              # dataclasses/enums and validation errors
  config.py                              # precedence, TOML/JSON loading, role/fallback resolution
  capabilities.py                        # probes and qualified capability records
  adapters.py                            # shared adapter protocol and registry
  dispatch.py                            # parallel read-only fan-out and result collection
  context.py                             # bounded packets, redaction, private prompt files
  sessions.py                            # exact session registry, lineage, context digests, usage
  leases.py                              # single-writer lease state
  audit.py                               # pre/post git, path, process, and patch audit
  setup.py                               # setup/doctor inventory and preference generation
```

This is a target decomposition, not a mandate to create every file immediately. The implementer must
survey size and coupling before splitting. Keep the entry point thin; no runtime module should become
an everything-bag. If a proposed module duplicates `workspace_guard.py`, `preflight_worktree.py`, or
`elves_landing_check.py`, centralize or call the existing utility instead.

Suggested operator commands (names may be tightened during Batch 1 without changing semantics):

```text
python3 scripts/cobbler_agents.py setup
python3 scripts/cobbler_agents.py doctor [--json]
python3 scripts/cobbler_agents.py validate-config
python3 scripts/cobbler_agents.py council --phase planning --packet <path>
python3 scripts/cobbler_agents.py session list|probe|resume
python3 scripts/cobbler_agents.py worker prepare|run|audit|export
```

Subprocess execution must use argv arrays with `shell=False`. Prompt delivery should use stdin or
mode-0600 files under `.elves/runtime`, never interpolate untrusted text into shell commands.

## Initial role configuration for this experiment

The committed Survival Guide, not public defaults, records these personal routes:

- **Planning council:** host coordinator + persistent Fable + persistent Grok parent + persistent
  Fugu Ultra, launched concurrently and read-only.
- **Supervision:** the current `gpt-5.6-sol` Ultra host owns contracts, risk decisions, disputed
  findings, acceptance, synthesis, and canonical run memory.
- **Implementation:** exact persistent Grok successor `9927883a-0203-42e1-a3e4-710a02096d46`
  receives one substantial complete batch per turn and targets two to five meaningful detached
  commits when natural. The host remains responsible for task packets, binary-patch import, run
  documents, branch git, validation, and synthesis. After three distinct Grok recovery attempts
  with no safe workaround, stop for the user; do not silently switch to Claude Code or host
  implementation.
- **Lightweight review:** ephemeral read-only `gpt-5.6-luna` at low reasoning may check a bounded
  diff, commit subject, or routine invariant. It cannot decide high-risk findings or write/push.
- **Validation:** host coordinator runs every local command and records evidence.
- **Review council:** fresh host-native reviewer + persistent Fable + persistent Fugu Ultra in
  parallel. The Grok implementer does not review its own patch.
- **Synthesis and run documents:** Sol host coordinator only.
- **Git/PR operations:** deterministic host shell commands only; no model invocation is required.

## Scope

### In Scope

- A generic external-harness contract and project-local role/fallback configuration.
- A provider-neutral lightweight-review role so a supervising host can delegate cheap bounded checks
  without changing the main run model; personal model IDs remain in the active Survival Guide only.
- Built-in adapters for Claude Code, Grok Build, and `codex-fugu`/Fugu Ultra using subscription or
  local auth where available.
- A custom CLI adapter contract suitable for user wrappers and future tools.
- Parallel read-only planning/review council dispatch with bounded reports and graceful fallback.
- Exact persistent sessions, context rehydration, lineage validation, observed-usage accounting,
  and honest unknown quota reporting.
- A single-writer external implementation lease, post-turn audit, and host-owned patch integration.
- A setup/doctor flow plus Claude `/setup-cobbler` and `/setup-council` managed aliases.
- Codex natural/skill invocation wording that does not claim unsupported top-level slash commands.
- OpenRouter, API-only models such as Muse, Codex-via-Claude wrappers, and future Google/Antigravity
  tools as optional custom/read-only recipes—not required built-ins.
- The coordinator-to-implementer clarity standard across core skill, templates, review, consistency
  checks, and tests.
- Final live qualification, master CouncilElves launch prompt, release docs, and local install checks.

### Out of Scope

- Storing, fetching, printing, or rotating API keys and subscription credentials.
- Building a hosted router, daemon, terminal UI, or universal provider SDK.
- Promising that every vendor supports persistent sessions, repo writes, arbitrary model selection,
  quotas, or worktrees.
- Making external tools necessary for normal Elves, Cobbler, Council aliases, or math workflows.
- Letting multiple external writers edit simultaneously.
- Letting a worker create/move refs, push, merge, open/own the PR, write run memory, or self-certify
  acceptance. A qualified detached commit chain under an explicit lease is permitted handoff state.
- Direct OpenRouter HTTP integration in the first release unless the runtime survey proves it can be
  added without a new dependency or secret-handling expansion. A custom wrapper recipe is sufficient.
- Automatic upgrade to newly released models. Doctor may discover and recommend; the user chooses.
- Copying any vendor's model identity, persona, policy, or safety prompt.

- tracked `references/models.toml.example` schema/example plus explicit documentation that the

- Do not make model names capability identifiers.
- Do not represent “installed” as “authenticated” or “advertised flag” as “qualified behavior.”
- Do not build dispatch/session/write logic into Batch 1 merely to make the CLI look complete.
- Do not let `scripts/cobbler_agents.py` become the long-term implementation module.

### Batch 2: Parallel read-only council dispatcher

**Intent:** Make Cobbler's planning and review fan-out real while preserving the current one-fitted-
answer product. This batch is read-only by construction; it does not yet support implementation
writes or session reuse beyond adapter invocation metadata.

**Build on:** Batch 1 contracts, `references/council-workflow.md` independence invariant,
`references/council-prompts.md` report schema, and the host-native fallback already documented.

**Expected surfaces:** focused runtime modules for context/redaction, subprocess execution, adapter
commands, and dispatch; fake CLI fixtures; technical reference updates. Keep provider command
construction inside adapters, not in the dispatcher.

**Tasks:**

- [ ] Build context packets from canonical disk state with task, role, mode, scope, relevant files,
      plan/run pointers, current date/HEAD, output schema, evidence needs, and forbidden actions.
- [ ] Redact prompt content and construct a minimal child environment. Preserve only necessary
      runtime/auth discovery variables; strip API-key/token/cloud/git credentials by default and
      report names stripped, never values.
- [ ] Use `asyncio.create_subprocess_exec` or equivalent argv-safe concurrency to start independent
      lanes together. Record launch/start/end timestamps, timeout, exit code, stdout/stderr summary,
      requested/actual model, and fallback result per lane.
- [ ] Implement Claude Code, Grok Build, Fugu, host-native, and custom-CLI **read-only** command
      builders with structured-output parsing and explicit tool/sandbox scope.
- [ ] Support a single bounded `lightweight_review` dispatch independently of council quorum. It may
      use a cheaper profile selected by config, must remain ephemeral/read-only, and cannot close a
      high-risk review or perform git/PR mutations.
- [ ] Store private prompt/result artifacts under ignored `.elves/runtime/council/<run-id>/` with
      restrictive permissions. Do not commit transcripts by default.
- [ ] Validate role-report schema and pass only successful bounded reports to host synthesis. One
      optional lane failure must not erase successful independent evidence.
- [ ] Add an explicit quorum/fallback policy. `target_quorum` is advisory by default: if optional
      fallbacks still leave fewer successful independent reports, continue with host synthesis,
      record the confidence drop, and do not label the result council-verified. `required_quorum`
      is valid only when that phase has `required = true` in the project Survival Guide; it counts
      successful independent reports (including a fresh host lane), tolerates individual optional
      lane failures while the count is met, and blocks after recovery/fallback when it is not met.

**Forbidden shortcuts and likely pitfalls:**

- Do not launch lanes sequentially and call it a council.
- Do not let roles see peer reports before synthesis.
- Do not interpolate task text into a shell command or inherit the full parent environment.
- Do not mark inference successful solely because the process exit code is zero; validate structured
  output, actual model, and terminal status.
- Do not treat unrelated MCP startup warnings as model inference failures without evidence.
- Do not cancel every successful lane because one optional adapter fails.

**Acceptance criteria:**
